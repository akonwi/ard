Ard provides a robust error handling system based on Result and Maybe types and the `try` keyword. This system encourages explicit error handling while providing convenient syntax for error propagation and transformation.

## Result Types

Result types represent operations that can either succeed with a value or fail with an error. They are written as `ValueType!ErrorType`:

```ard
fn divide(a: Int, b: Int) Int!Str {
  match b == 0 {
    true => Result::err("division by zero"),
    false => Result::ok(a / b),
  }
}
```

### Creating Results

Use the built-in constructors to create Result values:
- `Result::ok(value)` - creates a successful result
- `Result::err(error)` - creates an error result

### Working with Results

Results can be handled using pattern matching:

```ard
use go:fmt

fn divide(a: Int, b: Int) Int!Str {
  match b == 0 {
    true => Result::err("division by zero"),
    false => Result::ok(a / b),
  }
}

let result = divide(10, 2)
match result {
  ok(value) => fmt::Println("Result: {value}"),
  err(message) => fmt::Println("Error: {message}"),
}
```

Or use the `.or(default)` method to provide a fallback value:

```ard
fn divide(a: Int, b: Int) Int!Str {
  match b == 0 {
    true => Result::err("division by zero"),
    false => Result::ok(a / b),
  }
}

let safe_result = divide(10, 0).or(-1)  // Returns -1 on error
```

### Result Combinators

For transformation-heavy code, use combinators instead of nested `match` expressions:

- **`map`** — transforms the ok value; the callback returns a **plain value** that is automatically wrapped in `ok(...)`. Use when the transformation cannot fail.
- **`map_err`** — same idea but for the error side.
- **`and_then`** — the callback returns a **Result** itself, so it can succeed or fail. Use when the next step can produce its own error.

```ard
let res: Int!Str = Result::ok(21)

// map: always succeeds — callback returns a plain value
let doubled = res.map(fn(v) { v * 2 })          // ok(42)

// and_then: can fail — callback returns a Result
fn ensure_positive(n: Int) Int!Str {
  match n > 0 {
    true => Result::ok(n),
    false => Result::err("not positive"),
  }
}
let checked = doubled.and_then(ensure_positive)  // ok(42)

// map_err: transform the error side
let normalized = res.map_err(fn(err) { "calc failed: {err}" })
```

If inference is ambiguous, explicit type args can guide the checker:

```ard
fn parse_num(raw: Str) Int!Str {
  Result::ok(42)
}

let text = parse_num("42").and_then<Str>(fn(v) { Result::ok("{v}") })
```

## Maybe Types

Maybe types represent values that may or may not exist. They are written as `Type?` and are equivalent to `Maybe<Type>`:

```ard

struct User {
  name: Str,
  id: Int,
}

fn find_user(id: Int) User? {
  match id == 42 {
    true => Maybe::new(User{name: "Alice", id: 42}),
    false => Maybe::new(),
  }
}
```

### Creating Maybe Values

Use the standard library functions to create Maybe values:
- `Maybe::new(value)` - creates a value that exists
- `Maybe::new()` - creates an absent value

### Working with Maybe Values

Maybe values can be handled using pattern matching:

```ard
use go:fmt

struct User {
  name: Str,
  id: Int,
}

fn find_user(id: Int) User? {
  match id == 42 {
    true => Maybe::new(User{name: "Alice", id: 42}),
    false => Maybe::new(),
  }
}

let user = find_user(42)
match user {
  user => fmt::Println("Found user: {user.name}"),
  _ => fmt::Println("User not found"),
}
```

### Maybe Combinators

- **`map`** — transforms a present value; the callback returns a **plain value** that is automatically wrapped as present. Use when the transformation always produces a value.
- **`and_then`** — the callback returns a **Maybe** itself, so it can produce `Maybe::new(value)` or `Maybe::new<T>()`. Use when the next step might not produce a value.

```ard

// map: always produces a value — callback returns a plain value
let maybe_name: Str? = Maybe::new("Alice")
let maybe_len = maybe_name.map(fn(name) { name.size() }) // some(5)

// and_then: can produce none — callback returns a Maybe
fn non_empty(name: Str) Str? {
  match name.is_empty() {
    true => Maybe::new(),
    false => Maybe::new(name),
  }
}
let checked = maybe_name.and_then(non_empty) // some("Alice")
let empty: Str? = Maybe::new("")
let missing = empty.and_then(non_empty) // none
```

As with Results, explicit type args can guide inference when needed:

```ard
let maybe_text = Maybe::new(42).map<Str>(fn(v) { "{v}" })
```

## The try Keyword

The `try` keyword provides early return semantics for working with Result and Maybe types, eliminating the need for explicit error checking in many cases.

### With Result Types

```ard
fn divide(a: Int, b: Int) Int!Str {
  match b == 0 {
    true => Result::err("division by zero"),
    false => Result::ok(a / b),
  }
}

fn calculate() Int!Str {
  let x = try divide(10, 2)  // If divide fails, return early with error
  let y = try divide(x, 3)   // If this fails, return early with error
  Result::ok(y + 1)          // If we get here, both operations succeeded
}
```

### With Maybe Types

```ard

struct User {
  name: Str,
  id: Int,
}

fn find_user(id: Int) User? {
  match id == 42 {
    true => Maybe::new(User{name: "Alice", id: 42}),
    false => Maybe::new(),
  }
}

fn get_user_name(id: Int) Str? {
  let user = try find_user(id)  // If user not found, return early with none
  Maybe::new(user.name)        // If we get here, user was found
}

fn get_user_display(id: Int) Str? {
  let name = try get_user_name(id)  // Str? -> Str?, compatible types
  Maybe::new("User: {name}")
}
```

### Restrictions

- `try` can only be applied to Result or Maybe types
- When used without a catch clause, the function must return a compatible Result or Maybe type
- For Result types: the function must return a Result with the same error type
- For Maybe types: the function must return a Maybe type (inner types can be different)

## Error Transformation with Catch Blocks

When your function's return type doesn't match the Result's error type or when you want to handle Maybe `none` cases, you can use catch blocks:

```ard
fn parse_number(raw: Str) Int!Str {
  match raw == "abc" {
    true => Result::err("not a number"),
    false => Result::ok(42),
  }
}

fn process_data() Str {
  let num = try parse_number("abc") -> err {
    "Failed to parse: {err}"
  }
  "Number is: {num}"
}
```

### With Maybe Types

Use `_` in the catch clause because a `none` case carries no payload:

```ard

struct User {
  name: Str,
  id: Int,
}

fn find_user(id: Int) User? {
  match id == 42 {
    true => Maybe::new(User{name: "Alice", id: 42}),
    false => Maybe::new(),
  }
}

fn get_user_display(id: Int) Str {
  let user = try find_user(id) -> _ {
    "Unknown User"
  }
  "User: {user.name}"
}
```

### Catch Block Semantics

The catch block's value is **returned early from the enclosing function**, so its type must match the function's return type.

#### For Result Types
- **Success case**: The Result is unwrapped and execution continues normally
- **Error case**: The catch block executes and its result is returned early from the function

#### For Maybe Types
- **Some case**: The Maybe is unwrapped and execution continues normally
- **None case**: The catch block executes and its result is returned early from the function

### Function Reference Shorthand

For simple Result error transformations, you can reference a function directly. The function receives the error value:

```ard
fn risky_operation() Str!Str {
  Result::ok("data")
}

fn format_error(msg: Str) Str {
  "Error: {msg}"
}

fn process() Str {
  let result = try risky_operation() -> format_error
  "Success: {result}"
}
```

This is equivalent to:

```ard
fn risky_operation() Str!Str {
  Result::ok("data")
}

fn format_error(msg: Str) Str {
  "Error: {msg}"
}

fn process() Str {
  let result = try risky_operation() -> err {
    format_error(err)
  }
  "Success: {result}"
}
```

The shorthand applies to Result types, where the handler receives the error value. For Maybe types, use a `-> _ { ... }` catch block since `none` has no payload to pass.

## Unwrapping Nested Maybes

When working with nested structures where intermediate values are Maybe types, unwrap each level with `try`:

```ard

struct Profile {
  name: Str,
  bio: Str?,
}

struct User {
  id: Int,
  profile: Profile?,
}

fn get_profile_name(user: User?) Str {
  let u = try user -> _ { "no user" }
  let profile = try u.profile -> _ { "no profile" }
  profile.name
}
```

Each `try` unwraps one level; if any level is `none`, its catch block returns early with a fallback.

## Complete Example

Here's a comprehensive example showing different error handling patterns:

```ard
use go:fmt

struct Person {
  name: Str,
  age: Int,
}

struct Database {
  users: [Int: Person],
}

fn parse_age(s: Str) Int!Str {
  // Simulate parsing that can fail
  match s {
    "invalid" => Result::err("not a number"),
    _ => Result::ok(25),
  }
}

fn create_person(name: Str, age_str: Str) Person!Str {
  let age = try parse_age(age_str)  // Early return on parse error
  Result::ok(Person{name: name, age: age})
}

fn safe_create_person(name: Str, age_str: Str) Person {
  let age = try parse_age(age_str) -> err {
    fmt::Println("Age parsing failed: {err}")
    Person{name: name, age: 0}  // Early return with a default person
  }
  Person{name: name, age: age}
}

fn format_parse_error(msg: Str) Str {
  "Invalid age provided: {msg}"
}

fn create_person_with_message(name: Str, age_str: Str) Str {
  let person = try create_person(name, age_str) -> format_parse_error
  "Created person: {person.name}, age {person.age}"
}

// Maybe type examples
fn find_person(db: Database, id: Int) Person? {
  db.users.get(id)
}

fn get_person_info(db: Database, id: Int) Str? {
  let person = try find_person(db, id)  // Early return none if not found
  Maybe::new("Name: {person.name}, Age: {person.age}")
}

fn get_person_display(db: Database, id: Int) Str {
  let info = try get_person_info(db, id) -> _ {
    "Person not found"
  }
  "Person Info: {info}"
}
```

## Best Practices

### 1. Use try for Happy Path Code

The `try` keyword is designed to let you write code that focuses on the success case:

```ard
fn read_file(path: Str) Str!Str { Result::ok("{}") }
fn parse_json(content: Str) Str!Str { Result::ok(content) }
fn transform_data(parsed: Str) Str!Str { Result::ok(parsed) }

fn process_file() Str!Str {
  let content = try read_file("data.txt")
  let parsed = try parse_json(content)
  let result = try transform_data(parsed)
  Result::ok(result)
}
```

### 2. Transform Errors Appropriately

Use catch blocks to convert errors into appropriate types for your context:

```ard
fn internal_operation() Str!Str {
  Result::err("code 500")
}

fn user_friendly_operation() Str {
  try internal_operation() -> err {
    "Sorry, something went wrong. Please try again."
  }
}
```

### 3. Propagate Errors When Possible

When functions can naturally propagate errors, prefer `try` without catch:

```ard
fn first_operation() Int!Str { Result::ok(1) }
fn second_operation(input: Int) Int!Str { Result::ok(input + 1) }
fn third_operation(input: Int) Int!Str { Result::ok(input + 1) }

fn chain_operations() Int!Str {
  let step1 = try first_operation()
  let step2 = try second_operation(step1)
  let step3 = try third_operation(step2)
  Result::ok(step3)
}
```

## Error Handling vs Exceptions

Ard's error handling system differs from exception-based systems:

- **Explicit**: Errors are part of the type system and must be handled explicitly
- **Predictable**: No hidden control flow - errors only propagate through `try`

This approach encourages writing robust code while maintaining expressiveness.
