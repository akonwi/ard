# Ard Dependency System

## Summary

Design a Zig-inspired dependency system for Ard that fetches directly from git into a global cache, supports transitive dependencies, and allows multiple versions of the same dependency when they do not create real conflicts.

## Goals

- No central registry required
- Global cache for fetched dependencies
- Direct git-based dependencies
- Cryptographic integrity verification
- Support multiple versions of the same dependency
- Only surface conflicts when modules actually interact in incompatible ways
- Keep the mental model simple and explicit

## Core Philosophy

- **Zig-inspired**: No registry + global cache + direct from git
- **Multiple versions allowed**: Only conflict when modules actually interact
- **Explicit control**: Developer chooses when there are real conflicts
- **Content-addressable**: Cryptographic hashes ensure integrity

## Resolution Algorithm

### 1. Dependency Graph Building

```rust
fn resolve_dependencies(root_manifest: Manifest) -> Result<ResolutionResult, Error> {
    let mut resolved = HashMap::new();
    let mut to_process = VecDeque::from(root_manifest.dependencies);

    while let Some(dep) = to_process.pop_front() {
        let cached_dep = fetch_or_cache(dep)?;
        let dep_manifest = parse_manifest(&cached_dep)?;
        let namespace = create_namespace(&dep.name, &dep.version, &cached_dep.hash);

        resolved.insert(namespace, ResolvedDependency {
            name: dep.name,
            version: dep.version,
            hash: cached_dep.hash,
            manifest: dep_manifest,
        });

        for transitive in dep_manifest.dependencies {
            to_process.push_back(transitive);
        }
    }

    Ok(ResolutionResult { resolved })
}
```

### 2. Conflict Detection

```rust
fn detect_conflicts(resolved: &ResolutionResult) -> Vec<RealConflict> {
    let mut conflicts = Vec::new();

    for module in project.modules() {
        let direct_imports = module.direct_dependency_imports();
        let version_groups = direct_imports.group_by_name();

        for (dep_name, versions) in version_groups {
            if versions.len() > 1 {
                conflicts.push(RealConflict::DirectMultipleVersions {
                    module: module.name(),
                    dependency: dep_name,
                    versions: versions,
                });
            }
        }
    }

    conflicts
}
```

## Real Conflict Handling

```text
$ ard build
✗ real conflict detected in src/api.ard:

  module imports multiple versions of json:
    - json v1.0.0 (line 3: use json/parse)
    - json v2.1.0 (line 4: use json/serialize)

  error: cannot mix json_v1::Object and json_v2::Object types

  suggestions:
    1. Choose one version of json for this module
    2. Create separate modules for different versions
    3. Use explicit version namespaces:
       use json_1_0_0/parse
       use json_2_1_0/serialize
```

## Security Model

### Content Verification

- All dependencies verified by SHA-256 hash before use
- Hashes stored in lock file for reproducible builds
- Cache integrity checks on fetch
- Optional signature verification for trusted publishers

### Network Security

- HTTPS for all network requests
- SSH key support for private repositories
- Optional proxy/mirror support for corporate environments
- Audit log of all fetched dependencies

## Migration Strategy

### From Current System

- Automatic detection of existing `use project/module` imports
- Migration tool: `ard migrate` to convert to new format
- Compatibility mode during transition period
- Clear migration guide and examples

### For New Projects

- `ard new` creates projects with new dependency format
- Template projects showing best practices
- Good defaults for common scenarios

- ✅ Simple
- ✅ Powerful
- ✅ Secure
- ✅ Practical
